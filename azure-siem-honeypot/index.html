// 00 — EXECUTIVE SUMMARY

What Was Built

As part of an intensive cybersecurity training program, a live cloud honeypot was architected and deployed using Microsoft Azure. A Windows VM was intentionally exposed to the open internet to attract automated brute-force attacks. Using the Azure Monitor Agent (AMA) and Microsoft Sentinel SIEM, live security event logs were ingested and transformed into an interactive global threat intelligence dashboard.

⚠
Disclaimer: This environment was intentionally misconfigured for educational research only. Microsoft Defender was disabled and all NSG rules were opened. Never replicate in production.

// 02 — INFRASTRUCTURE DEPLOYMENT

Step-by-Step Build

Every step from zero to a fully operational honeypot environment.

STEP 01

Resource Group & Virtual Network

Created a dedicated Resource Group to logically isolate all honeypot resources. Provisioned a Virtual Network (VNet) to define the network boundary and enable controlled internal routing.

Resource GroupVNet
STEP 02

Windows 10 Virtual Machine

Deployed a Windows 10 VM inside the VNet. This machine acts as the primary bait target — sitting exposed on the internet, waiting for automated scanners and brute-force bots to discover it.

Windows 10Azure VM
STEP 03

NSG Inbound Rule — Allow All Traffic

Configured a custom Network Security Group (NSG) inbound rule with wildcard permissions (* : *) across all ports and protocols. This disabled the Azure firewall, opening RDP port 3389 directly to the public internet.

↑ NSG Inbound Security Rules — nomipc-yt-nsg showing Allow All configuration

NSGRDP 3389All Traffic
STEP 04

Disabled Microsoft Defender on the VM

Turned off Microsoft Defender on the honeypot VM to prevent it from blocking incoming attack attempts, ensuring clean and unfiltered Windows Security Event logs would flow through.

⚠
Disabling Defender exposes the host to real malware. This is intentional in a honeypot lab context only. Never disable Defender on a production or personal machine.

// 04 — SIEM DASHBOARD ENGINEERING

Sentinel Workbook & KQL Analytics

Raw Windows Event logs transformed into actionable SOC-ready monitoring widgets inside a custom Sentinel Workbook.

Widget A — Top Attacker IPs & Usernames

KQL queries were written to isolate Event ID 4625 and extract the IP addresses and specific account names targeted by bots. This revealed a heavy reliance on default credentials such as admin and administrator.

// Failed login attempts — Event ID 4625 SecurityEvent | where EventID == 4625 | summarize count() by IpAddress, TargetUserName | order by count_ desc | take 20

↑ Widget A — Top attacker IPs (8.17k from 80.94.95.83) and most-targeted usernames (\ADMINISTRATOR, \ADMIN)

Widget B — Failed Login Attempts Over Time (Event ID 4625)

This widget visualizes the volume of Event ID 4625 (Failed Login Attempts) over time, revealing attack patterns, peak hours, and the sheer scale of automated scanning activity targeting the exposed honeypot VM.

ℹ
Event ID 4625 — An account failed to log on. This is the primary indicator of brute-force activity and the core metric tracked in this honeypot lab.
// Failed login attempts over time — Event ID 4625 SecurityEvent | where EventID == 4625 | project TimeGenerated, IpAddress, Computer, Account, Activity, IpPort | summarize Attacks=count() by bin(TimeGenerated, 30m) | order by TimeGenerated desc

↑ Widget B — 112+ attack bursts per interval, peaking after 9:30 PM as bots ramped up

// 05 — GLOBAL THREAT INTELLIGENCE

Live Heat Map Visualization

Microsoft Sentinel's native geo_info_from_ip_address() function was used to extract latitude, longitude, and country from attacker IPs — with no external API keys required.

// Geo-enriched attacker map data SecurityEvent | where EventID == 4625 | extend GeoInfo = geo_info_from_ip_address(IpAddress) | extend Latitude = toreal(GeoInfo.latitude), Longitude = toreal(GeoInfo.longitude), Country = tostring(GeoInfo.country) | summarize Attacks=count() by Latitude, Longitude, Country | render map

↑ Live Sentinel Heat Map — bubble size = attack volume. Heavy concentration from Europe & Asia

ℹ
Within hours of dropping the NSG firewall, global botnets discovered the open RDP port and commenced automated brute-force operations — demonstrating the speed and volume of internet background noise targeting any exposed cloud infrastructure.
